# llm-wiki-agent

Учебный репозиторий кэпстоун-проекта для курса «Разработка AI-агентов».

`llm-wiki-agent` — это учебный **агент корпоративной LLM Wiki**: учебная версия
живой LLM Wiki о компании как системе. В качестве учебного мира используется
**Northstar Compute** — вымышленная AI-инфраструктурная компания в духе
современных GPU-облаков и датацентровых бизнесов. Она продаёт управляемую
GPU-capacity для обучения и инференса, спорит о reserved-контрактах, переживает
инциденты кластеров, считает маржу и конкурирует за AI-команды.

Домен специально вымышленный. Для сдачи важно не знать рынок GPU-облаков, а
уметь проследить путь `raw → wiki → context → answer` и объяснить, почему
харнес выбрал именно этот контекст.

В полном замысле такая wiki копит и связывает знания о команде, процессах,
рынке, конкурентах, нише, бизнес-модели, метриках и решениях. В учебном
базовом срезе корпус маленький, но архитектура остаётся той же: raw-источники,
производный wiki-слой и схема с правилами поддержки.

Проект вдохновлён идеей Андрея Карпатого про
[LLM Knowledge Bases](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f),
но не пытается копировать чужой рабочий процесс. Для курса важнее инженерный путь:
контекст, инструменты, трейс, состояние запуска, проверка состояния, схема и границы
доверия.

## Идея LLM Wiki

Это не просто хранилище документов и не обычный RAG поверх папки файлов. Идея
в том, что агент постепенно поддерживает **накапливающийся артефакт знаний**:
читает новые источники, обновляет связанные страницы, замечает противоречия,
ведёт индекс и оставляет журнал изменений.

В проекте есть три слоя:

1. **Raw-источники** — исходные материалы в `vault/raw/`. Это источник истины:
   агент читает их, но не переписывает.
2. **Wiki** — производный слой в `vault/wiki/`: страницы источников, концептов,
   связей, сравнений и будущих синтезов.
3. **Схема** — правила устройства wiki и рабочих процедур в
   `docs/wiki-schema.md`. Она объясняет агенту, как пополнять wiki, как
   отвечать на вопросы и как не смешивать устойчивое знание с артефактами одного
   запуска.

## Учебный Мир

Northstar Compute специально сделана не «складом документов», а маленькой
моделью живого AI-инфраструктурного бизнеса. В исходном корпусе есть:

- базовый бриф компании;
- заметка по планированию capacity на квартал;
- звонок с AI-лабораторией Lumen Labs;
- инцидент Aurora-17;
- заметка про pricing и маржу;
- обзор рынка и конкурентов.

Эти источники дают студенту достаточно материала, чтобы wiki связывала
продуктовые обещания, GPU-capacity, клиентские ожидания, диагностику инцидентов
и рыночную позицию компании. Лор вымышленный: он вдохновлён общими паттернами
рынка AI-инфраструктуры, но не копирует реальные внутренние документы Nebius,
CoreWeave или других компаний.

## Быстрый Старт

```bash
cd llm-wiki-agent
uv sync --frozen --extra dev
uv run --with pytest python -m pytest
uv run kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault
uv run kb-eval --eval-fixture fixtures/evals/cases.json --vault-root vault
```

Локальный демо-запуск работает без API-ключа. Это детерминированный режим на
фикстурах для тестов, CI, офлайн-работы и сравнения.

| Команда | Что делает | Когда запускать | Успешный результат |
| --- | --- | --- | --- |
| `uv sync --frozen --extra dev` | ставит зависимости через `uv` | один раз после клонирования | зависимости установлены без ошибок |
| `uv run --with pytest python -m pytest` | запускает тесты проекта | перед сдачей и после правок | `passed` в выводе pytest |
| `uv run kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault` | собирает локальный wiki-путь без API-ключа | чтобы увидеть артефакты запуска | появился `run_id`, ответ, сводка и отчёт проверки состояния |
| `uv run kb-eval --eval-fixture fixtures/evals/cases.json --vault-root vault` | запускает маленький eval-набор | перед сдачей кэпстоуна | `status: pass`, `passed_count == case_count` |
| `rm -rf artifacts vault/wiki vault/outputs vault/index.md vault/log.md` | сбрасывает только производные файлы | если нужно начать локальный прогон заново | остаётся только исходный `vault/raw/` |
| `OPENAI_API_KEY=... uv run --extra openai kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault --live-openai` | делает финальный ответ через OpenAI Responses API | опционально, если есть ключ | появился `artifacts/responses/<run_id>.json` |

Live-запуск:

```bash
OPENAI_API_KEY=... uv run --extra openai kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault --live-openai
```

Live-запуск проходит тот же путь, но финальный Markdown-ответ собирается через
OpenAI Responses API. Модель получает только выбранный вопрос, wiki-контекст и
исходное основание. Она не получает права менять `vault/raw/` или `vault/wiki/`.

## Что Создаёт Запуск

Артефакты для чтения:

- `vault/index.md`
- `vault/log.md`
- `vault/wiki/index.md`
- `vault/wiki/sources/<source_id>.md`
- `vault/wiki/concepts/<concept_id>.md`
- `docs/wiki-schema.md`
- `vault/outputs/<run_id>.md`
- `vault/outputs/<run_id>-summary.md`

Артефакты запуска:

- `artifacts/runs/<run_id>.json`
- `artifacts/traces/<run_id>.jsonl`
- `artifacts/plans/<run_id>.json`
- `artifacts/context/<run_id>.json`
- `artifacts/tools/<run_id>.json`
- `artifacts/health/<run_id>.json`
- `artifacts/evals/<eval_run_id>.json` после eval-запуска
- `artifacts/responses/<run_id>.json` только для live-запуска

`artifacts/responses/` хранит метаданные ответа: `response_id`, модель, usage и
длину ответа. Полный Markdown-ответ лежит в `vault/outputs/`.

## Мини-словарь

- **Трейс** — журнал событий запуска в `artifacts/traces/`.
- **Пакет контекста** — JSON, где видно, какие wiki- и raw-материалы попали в
  ответ.
- **Запись запуска** — JSON с состоянием конкретного запуска.
- **Отчёт проверки состояния** — проверка, что ключевые артефакты запуска на месте.
- **Eval-отчёт** — маленькая проверка привязки к источникам и сценариев сбоя.
- **Схема** — правила структуры wiki и процедур её поддержки.

## Что Сейчас Реализовано

Текущий каркас покрывает базовый срез `M0 + M1 + M2 + M3` на корпусе Northstar
Compute:

- сбор wiki-слоя из локального корпуса;
- явное разделение трёх слоёв: raw-источники, wiki и схема;
- путь ответа `index → concepts → sources → raw`;
- план ответа перед финальной генерацией;
- пакет контекста и лестницу решений;
- реестр контрактов инструментов с разделением режима чтения и записи;
- запись запуска, JSONL-трейс и отчёт проверки состояния;
- сводку запуска, которую удобно открыть первой;
- mini-eval на несколько сценариев привязки к источникам и сбоя;
- live-путь ответа через OpenAI Responses API.

В базовом LMS-сценарии отдельного `M4` нет: финальная сдача строится вокруг
чтения одного `run_id`, артефактов `M2/M3`, eval-отчёта и короткой инженерной
заметки.

Проект остаётся системой, которая сначала учится безопасно читать. Запись в
устойчивый слой знаний появится только в отдельном расширении через предложение
изменения, предпросмотр и подтверждение.

## Зависимости

Базовый путь:

- Python `3.11`
- `uv`

Live-путь:

- `OPENAI_API_KEY`
- зависимость `openai`, которую live-команда подтягивает через
  `uv run --extra openai`

Brave Search API и X API не нужны для базовых вех.

## Как Читать Проект

- `vault/` — человекочитаемый слой знаний, ответы и сводки запусков.
- `artifacts/` — доказательства конкретного запуска.
- `docs/milestones/` — цель и критерии приёмки каждой вехи.
- `docs/final-project.md` — бриф, критерии приёмки и рубрика кэпстоун-проекта.
- `docs/company-lore.md` — учебный мир Northstar Compute и границы лора.
- `docs/wiki-schema.md` — схема wiki и правила поддержки живой базы знаний.
- `docs/trust-boundaries.md` — где проходит граница доверия.
- `docs/tool-contracts.md` — какие инструменты только читают, а какие пишут.
- `docs/runbook.md` — короткий маршрут диагностики одного запуска.
- `docs/handoff-template.md` — шаблон инженерной заметки к сдаче или вопросу.
- `docs/dependency-policy.md` — что входит в основной путь, а что остаётся расширением.
- `AGENTS.md` — рабочий контракт для студента и кодинг-агента.

Obsidian удобен для просмотра `vault/`, но не обязателен. Любой редактор
Markdown тоже подходит.

Если локальные артефакты накопились и мешают, запустите:

```bash
rm -rf artifacts vault/wiki vault/outputs vault/index.md vault/log.md
```

Эта команда не удаляет `vault/raw/`.

## Как просить о помощи

Если запуск не проходит, приложите к вопросу короткий debug-пакет:

- версию ОС и вывод `uv --version`;
- commit SHA или ссылку на репозиторий;
- точную команду, которая упала;
- первые строки ошибки;
- если `run_id` уже появился: `artifacts/runs/<run_id>.json`,
  `artifacts/health/<run_id>.json` и последние строки
  `artifacts/traces/<run_id>.jsonl`.

Не публикуйте `OPENAI_API_KEY`, `.env` и другие секреты. Если сомневаетесь,
сначала удалите из вывода всё, что похоже на ключ, токен или приватный URL.

Подробный чеклист отладки лежит в `docs/troubleshooting.md`, лёгкий runbook —
в `docs/runbook.md`, а карта расширений после базового среза — в
`docs/roadmap.md`.
