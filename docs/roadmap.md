# Карта расширений

Базовый кэпстоун-проект намеренно ограничен. Он проверяет, что студент умеет
собрать и прочитать один локальный агентный контур: raw-источники, wiki,
контекст, ответ, трейс, health и eval.

Ниже — расширения после базовой сдачи.

## Расширение A: контролируемая обратная запись

Агент не пишет в wiki молча. Правильный путь:

1. подготовить proposal;
2. показать diff или preview человеку;
3. сохранить решение в стейте запуска;
4. применить patch обычным кодом;
5. оставить событие в трейсе.

## Расширение B: regression harness

Добавить больше eval-кейсов:

1. вопрос с недостаточным контекстом;
2. конфликт между raw-источниками;
3. проверка provenance;
4. регрессия после изменения промпта;
5. проверка grounding для ответа по wiki.

## Расширение C: hosted path

Минимальный production-adjacent путь:

1. HTTP endpoint для запроса;
2. background runner;
3. health/readiness endpoint;
4. секреты через env;
5. smoke-test;
6. короткий rollback note.

## Расширение D: enterprise-слой

Темы для будущего продвинутого блока:

1. OAuth и scopes;
2. RBAC;
3. audit trail;
4. token audience;
5. границы данных между пользователями и организациями;
6. rate limits и retry semantics.

## Расширение E: оркестрация

Субагенты и хэндофы стоит добавлять только после того, как один агент уже
стал тесным, а evals показывают пользу разделения. Иначе orchestration
добавляет стоимость, задержку и поверхность отказов без выигрыша.
