# M3

В текущем `main` майлстоун `M3` уже реализован как учебный проверочный слой
управляемого запуска. Это не ветка, не тег и не задание «дописать
runtime с нуля», а карта чтения: студент берёт один `run_id` и связывает
запись запуска, трейс, health-отчёт, бюджет, tools и eval.

## Цель

В майлстоуне `M3` мы добавляем слой устойчивого
выполнения поверх уже готового пути `M2`.

Система должна показывать не только ответ, план и пакет контекста, но и то, как
конкретный запуск жил как объект: где он стартовал, чем завершился, какие
события оставил и проходит ли базовую проверку состояния.

## Стартовое состояние

- точка входа: текущий `main` после прохождения `M2`
- до начала уже должно работать:
  - `uv run --with pytest python -m pytest`
  - `uv run kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault`
  - `artifacts/plans/<run_id>.json`
  - `artifacts/context/<run_id>.json`
  - `artifacts/traces/<run_id>.jsonl`

## Объём

Нужно уметь увидеть в текущем проекте:

- хранить запись запуска как состояние запуска, а не только как случайный JSON;
- записывать `terminal_reason` при успешном завершении;
- сохранять реестр контрактов инструментов для текущих режимов чтения и записи;
- добавить отчёт проверки состояния для ключевых артефактов запуска;
- держать `run_budget.yaml` как читаемый бюджет по стадиям;
- показать студенту, что запуск можно диагностировать по трейсам, артефактам и
  отчёту проверки состояния.

Вне объёма:

- распределённая очередь
- настоящие фоновые worker-процессы
- production leases
- полноценный pause/resume/cancel runtime
- обратная запись в wiki
- UI подтверждения
- большой eval-каркас с LLM-judge или внешними сервисами

## Критерии приёмки

- [ ] `uv run --with pytest python -m pytest` проходит
- [ ] `uv run kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault` проходит
- [ ] запись запуска содержит `status: completed`
- [ ] запись запуска содержит `terminal_reason: success`
- [ ] отчёт проверки состояния создан и содержит `status: pass`
- [ ] eval-отчёт создан и содержит `status: pass`
- [ ] трейс содержит события планирования и ответа
- [ ] сводка запуска создана и ссылается на ответ, контекст, инструменты, трейс и отчёт проверки состояния
- [ ] реестр инструментов отделяет режим только для чтения от режима записи
- [ ] `run_budget.yaml` есть и описывает основные стадии запуска
- [ ] live-путь, если он использовался, пишет только ответ и метаданные
      как артефакты
      и не делает обратную запись в `vault/raw/` или `vault/wiki/`

## Подсказки для отладки

- если отчёт проверки состояния не прошёл, начните с отсутствующих проверок внутри
  `artifacts/health/<run_id>.json`;
- если запуск не завершён, сравните `artifacts/runs/<run_id>.json` и трейс;
- если трейс неполная, проверьте, на каком событии остановился запуск;
- если budget выглядит как приложение про pricing, рамка стала слишком широкой: здесь нужен
  бюджет запуска по стадиям.
