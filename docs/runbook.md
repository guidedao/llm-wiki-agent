# Лёгкий runbook

Runbook нужен не для продакшен-бюрократии, а чтобы студент быстро понял, где
сломался один запуск.

## Быстрый маршрут

1. Откройте `vault/outputs/<run_id>-summary.md`.
2. Если ответа нет, откройте `artifacts/runs/<run_id>.json`.
3. Если статус не `completed`, посмотрите последние строки
   `artifacts/traces/<run_id>.jsonl`.
4. Если статус `completed`, но ответ странный, откройте
   `artifacts/context/<run_id>.json`.
5. Если контекст выглядит правильным, проверьте `artifacts/health/<run_id>.json`.
6. Если локальный путь выглядит здоровым, запустите `kb-eval`.

## Частые симптомы

| Симптом | Первый файл | Вероятный слой | Что сделать |
| --- | --- | --- | --- |
| Нет `run_id` | вывод команды | CLI или входные пути | проверьте, что команда запущена из корня репозитория |
| Нет `vault/wiki/` | `vault/` | компиляция wiki | запустите `uv run kb-agent ...` заново |
| Ответ без источников | `vault/outputs/<run_id>.md` | grounding | сравните ответ с `artifacts/context/<run_id>.json` |
| Контекст не про вопрос | `artifacts/context/<run_id>.json` | ретривал или план | сравните план, выбранные wiki-страницы и raw-источники |
| Health вернул `fail` | `artifacts/health/<run_id>.json` | артефакты запуска | найдите проверку со статусом `fail` |
| Eval вернул `fail` | `artifacts/evals/<eval_run_id>.json` | набор проверок | откройте `failure_reason` у провалившегося кейса |
| Live-запуск упал | `artifacts/responses/` и терминал | OpenAI Responses API | сначала проверьте локальный deterministic-путь без `--live-openai` |

## Правило диагностики

Не начинайте с переписывания промпта. Сначала установите слой отказа:

1. входные файлы;
2. wiki-слой;
3. план;
4. пакет контекста;
5. финальный ответ;
6. трейс;
7. health/evals.

Только после этого меняйте код, промпт или корпус.
