# Кэпстоун-проект

Вы собираете учебного **агента корпоративной LLM Wiki**: учебную версию живой LLM Wiki о
компании как системе. Такая wiki нужна не для хранения пары файлов, а для
постепенного накопления связанной картины: команда, процессы, рынок,
конкуренты, ниша, бизнес-модель, метрики, решения и открытые вопросы.

Это не техническое собеседование и не попытка построить «всю корпоративную
память навсегда». Достаточно ограниченного среза такой живой базы знаний и
проверяемых артефактов.

## Сюжет

Вы агентный инженер во внутренней команде платформы и продуктивности **Northstar
Compute**. Это вымышленная AI-инфраструктурная компания: она строит GPU-облако
для training- и inference-нагрузок, продаёт reserved-capacity, держит
датацентровые кластеры, разбирает инциденты и конкурирует на рынке managed
GPU-infrastructure.

Вам поручили собрать прототип агента, который помогает компании поддерживать
обновляемую wiki о себе и своей среде:

1. исходные документы лежат в `vault/raw/`;
2. правила структуры и процедур описаны в `docs/wiki-schema.md`;
3. агент собирает производный wiki-слой в `vault/wiki/`;
4. на вопрос пользователя агент выбирает контекст и отвечает со ссылками;
5. каждый запуск оставляет `run_id`, трейс, пакет контекста, отчёт проверки состояния и
   сводку.

Важно: это не симулятор «разбери всю компанию навсегда». Учебный корпус
маленький и заранее подготовленный. В нём есть ровно столько напряжения, чтобы
агенту было что связывать: планирование capacity, клиентские обещания,
диагностику инцидентов, pricing, маржу и конкурентное позиционирование.

## Три слоя

1. **Raw-источники** — исходные материалы. Это подготовленный источник истины: заметки,
   документы, выдержки из встреч, рыночные наблюдения, метрики и другие входы.
2. **Wiki** — производный слой. Здесь появляются страницы источников,
   концептов, сущностей, сравнений, противоречий и синтезов.
3. **Схема** — правила поддержки wiki. Она задаёт структуру страниц,
   соглашения по ссылкам, порядок приёма источников, ответа на вопросы,
   проверки wiki и границы записи.

## Что Уже Есть В Корпусе

- `company-brief.md` — кто такая Northstar Compute и в чём её главный
  операционный конфликт.
- `capacity-planning-q2.md` — как команда разделяет выделенную capacity,
  burst-capacity и буфер.
- `customer-call-lumen-labs.md` — что важно клиенту при переносе training-run.
- `incident-aurora-17.md` — почему один статус очереди скрыл разные причины
  ожидания.
- `pricing-margin-memo.md` — как pricing связан с маржой и обещаниями клиенту.
- `market-competitors.md` — с кем компания конкурирует и как аккуратно
  формулирует позиционирование.

## Рабочий Сценарий

Представьте, что к вам приходит коллега из продуктовой команды или поддержки и спрашивает не
«что лежит в папке», а что компания уже знает по конкретному рабочему вопросу:

1. зачем Lumen Labs нужен capacity buffer;
2. почему инцидент Aurora-17 нельзя объяснять одним статусом очереди;
3. как pricing связан с маржой и обещаниями клиенту;
4. чем Northstar Compute отличается от других GPU-cloud альтернатив;
5. что система должна честно не отвечать, если в raw-источниках нет фактов.

Ваша задача — не сочинить лучший бизнес-ответ, а показать маленький
инженерный контур: исходное основание, wiki-страницы, пакет контекста, ответ с
опорой на источники, трейс, отчёт проверки состояния и eval.

## Что нужно сдать

Перед сдачей должны проходить:

```bash
uv sync --frozen --extra dev
uv run --with pytest python -m pytest
uv run kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault
uv run kb-eval --eval-fixture fixtures/evals/cases.json --vault-root vault
```

Если есть `OPENAI_API_KEY`, дополнительно можно показать:

```bash
OPENAI_API_KEY=... uv run --extra openai kb-agent --query-fixture fixtures/queries/m0_query.json --vault-root vault --live-openai
```

Live-запуск не обязателен для CI и не должен быть блокером, если у студента нет
ключа.

Если нужно сбросить локально сгенерированные файлы и повторить прогон:

```bash
rm -rf artifacts vault/wiki vault/outputs vault/index.md vault/log.md
```

Команда удаляет только производные артефакты и не трогает `vault/raw/`.

## Минимальный Рабочий Срез

- локальный markdown-корпус в `vault/raw/`;
- схема в `docs/wiki-schema.md`;
- сгенерированный wiki-слой в `vault/wiki/`;
- ответ в `vault/outputs/<run_id>.md`;
- сводка запуска `vault/outputs/<run_id>-summary.md`;
- `artifacts/context/<run_id>.json`;
- `artifacts/tools/<run_id>.json`;
- `artifacts/traces/<run_id>.jsonl`;
- `artifacts/health/<run_id>.json`;
- `artifacts/evals/<eval_run_id>.json`.

## Что проверять руками

Возьмите один ответ, например про Lumen Labs или Aurora-17, и разберите его как
маленький peer review:

1. откройте сводку запуска;
2. перейдите из неё в ответ;
3. проверьте раздел `## Цитаты`;
4. откройте пакет контекста и убедитесь, что выбранные raw-источники совпадают
   с ответом;
5. откройте реестр инструментов и проверьте, что инструменты записи не имеют
   `model_safe: true`.
6. откройте отчёт проверки состояния и eval-отчёт: оба должны иметь `status: pass`.

## Рубрика

| Критерий | Баллы |
| --- | ---: |
| Рабочий ограниченный сценарий: тесты, локальный демо-запуск, eval-запуск, локальные фикстуры, путь `raw -> wiki -> context -> answer` | 20 |
| Ответ с опорой на источники: источники, provenance, отсутствие выдуманных ссылок, честный отказ или abstain в eval | 20 |
| Проверяемость: план, запись запуска, JSONL-трейс, отчёт проверки состояния, сводка по `run_id` | 15 |
| Дисциплина wiki: не смешаны raw-источники, wiki, схема, retrieval, артефакты запуска и устойчивый слой знаний | 15 |
| Границы безопасности: разделение чтения и записи, нет автономной обратной записи, live-путь не пишет в `vault/raw/` или `vault/wiki/` | 10 |
| Мини-evals: 3-5 сценариев, включая недостаточный контекст, конфликт или вопрос по capacity/pricing/incident | 10 |
| Короткая инженерная заметка: что сделано, что вне объёма, как диагностировать один запуск | 10 |

## Что не нужно для базового среза

- hosted-deployment;
- enterprise auth/RBAC;
- multi-agent orchestration;
- автономная обратная запись;
- web crawling и внешние search/social API;
- полноценный threat model на много страниц;
- vector DB или OpenAI File Search как обязательный путь.

Эти темы можно добавить как расширение, когда базовый срез уже стабильно проходит.

## Расширение

Хорошее расширение после базового среза: контролируемая обратная запись.

Правильная форма:

1. агент готовит предложение изменения;
2. человек видит предпросмотр;
3. подтверждение сохраняется в состоянии запуска;
4. приложение применяет patch;
5. трейс показывает, кто и что разрешил.
